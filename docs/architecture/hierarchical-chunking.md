# Hierarchical Chunking Design

## Overview

This document outlines the design for hierarchical chunking, a future enhancement to improve retrieval quality for complex, multi-concept queries.

## Current State

CLORAG currently uses flat chunking with:
- Semantic-aware splitting (code blocks, tables, headings preserved)
- Token-based sizing (256-512 tokens per chunk)
- Voyage `contextualized_embed()` for context-aware embeddings
- Hybrid search (dense + BM25 sparse vectors)

This works well for single-concept queries but can lose document-level context for multi-concept questions.

## Proposed Architecture

### Three-Level Hierarchy

```
Level 0: Document Summaries
    │
    ├── Level 1: Section Summaries
    │       │
    │       └── Level 2: Paragraph Chunks (current)
```

### Level Descriptions

**Level 0 - Document Summaries**
- One summary per document (generated by Claude Haiku)
- Captures main topics, products mentioned, key concepts
- Size: ~100-150 tokens
- Used for broad topic matching

**Level 1 - Section Summaries**
- One summary per major section (## headings)
- Captures section purpose and key points
- Size: ~50-100 tokens
- Bridges document and chunk context

**Level 2 - Paragraph Chunks**
- Current chunking strategy (token-aware)
- Contains actual content for detailed answers
- Size: 350-450 tokens (configurable)

### Qdrant Schema Changes

```python
# New collection structure
{
    "dense": vectors.Distance.COSINE,
    "sparse": vectors.Distance.DOT,
}

# Extended metadata
{
    "hierarchy_level": 0 | 1 | 2,
    "parent_id": str | None,  # Level 1 → Level 0, Level 2 → Level 1
    "children_ids": list[str],  # For parent nodes
    "is_summary": bool,
    "document_id": str,  # Links all levels to source document
}
```

### Retrieval Flow

```
Query
  │
  ├── 1. Search Level 2 (chunks) → Get top-K chunks
  │
  ├── 2. Fetch Level 1 parents → Get section context
  │
  ├── 3. Fetch Level 0 parents → Get document context
  │
  └── 4. Rerank with hierarchy-aware scoring
```

### Hierarchy-Aware Reranking

```python
final_score = (
    0.6 * chunk_relevance +      # Level 2 score
    0.25 * section_relevance +   # Level 1 score
    0.15 * document_relevance    # Level 0 score
)
```

## Implementation Plan

### Phase 1: Schema Migration

1. Add `hierarchy_level` and `parent_id` fields to existing chunks (Level 2)
2. Backfill `document_id` for existing data
3. Create migration script for existing collections

### Phase 2: Summary Generation

1. Generate Level 0 summaries for existing documents using Haiku
2. Generate Level 1 summaries for sections
3. Embed and store summaries in same collection

### Phase 3: Retrieval Enhancement

1. Update `MultiSourceRetriever` to fetch hierarchy
2. Implement hierarchy-aware reranking
3. Add configuration for hierarchy weights

### Phase 4: Ingestion Pipeline Updates

1. Update `DocusaurusIngestionPipeline` to generate summaries
2. Update `CuratedGmailPipeline` to generate summaries
3. Update `CustomDocumentService` to generate summaries

## Performance Considerations

### Additional Latency

- Summary generation: ~200-500ms per document (one-time, during ingestion)
- Parent fetch: ~5-10ms per query (single Qdrant request)
- Reranking: Minimal impact (already doing cross-encoder reranking)

### Storage Overhead

- Level 0: ~1 vector per 10 chunks (10% overhead)
- Level 1: ~1 vector per 3 chunks (33% overhead)
- Total: ~43% more vectors

### Mitigation Strategies

1. Lazy summary generation (generate on first query if missing)
2. Cache hierarchical relationships in memory
3. Batch parent fetching across multiple chunks

## When to Implement

Consider implementing hierarchical chunking when:

1. Complex queries require understanding of multi-section context
2. Troubleshooting flows span multiple document sections
3. Product comparison queries need broader context
4. Current retrieval quality plateaus despite parameter tuning

### Quality Metrics to Monitor

- MRR (Mean Reciprocal Rank) on evaluation set
- User satisfaction scores on complex queries
- Support ticket escalation rates
- Response coherence ratings

## Alternative Approaches Considered

### Late Chunking (Jina)
- Pros: Single embedding pass, theoretically optimal
- Cons: Requires specific model (jina-embeddings-v3), 2048 token limit

### Full Document Embedding
- Pros: Simpler, captures full context
- Cons: Context length limits, expensive, loses precision

### Multi-Query Retrieval
- Pros: Can be added without schema changes
- Cons: Higher latency, more API calls

## Conclusion

Hierarchical chunking offers 25-35% improvement potential for complex queries based on 2025 RAG benchmarks. Implementation should be considered after optimizing current token-aware chunking and establishing baseline quality metrics.

The current implementation with token-aware chunking + Voyage contextualized embeddings + reranking provides a strong foundation. Hierarchical chunking is recommended as a Phase 2 enhancement once query complexity patterns are better understood from production usage.
