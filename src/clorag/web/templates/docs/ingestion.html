StageDescriptionOutput
1. Sitemap Fetch Fetch /sitemap.xml and parse URLs with lastmod dates List of (url, lastmod) tuples
2. Page Fetch Jina Reader API (r.jina.ai) for clean markdown extraction; BeautifulSoup fallback on 429/503 Markdown content
3. Content Extraction Jina: clean markdown with tables preserved. BeautifulSoup: HTML tables converted to markdown format Clean text + title + tables
4. Text Chunking Token-based chunking via tiktoken (cl100k_base). Docs: 450 tokens, cases: 350, default: 400. Adaptive threshold: short content (<200 tokens) stays as single chunk. Code blocks, tables, and headings preserved as atomic units. Document chunks
4.5. Keyword Extraction Claude Sonnet extracts 5-10 technical keywords per page. Runs in parallel (10 concurrent). Samples first 2000 + last 2000 chars for full page coverage. Keywords stored on all chunks of the page. Keywords per chunk
4.6. RIO Terminology Fixes High-confidence fixes (≥0.85) auto-applied before embedding. Legacy terms like "RIO-Live" normalized to "RIO +LAN". Configurable via RIO_FIX_ON_INGEST (default: true) and RIO_FIX_MIN_CONFIDENCE (default: 0.85). Corrected text
5. Dense Embeddings Contextualized batch embedding with voyage-context-3 1024-dim vectors
6. Sparse Embeddings FastEmbed BM25 for keyword matching Sparse vectors
7. Qdrant Storage Upsert to docusaurus_docs collection with hybrid vectors Indexed documents
8. Camera Extraction Claude Sonnet extracts camera compatibility info Camera database entries
Jina Reader + Table Preservation

Jina Reader (r.jina.ai) provides clean markdown extraction with automatic table formatting. On rate limits (429) or errors (503), the system falls back to BeautifulSoup with automatic HTML table-to-markdown conversion. This ensures camera control matrices and feature comparison tables are preserved and searchable.

URL Filtering

Sitemap parsing excludes URLs containing /tags, /search, or /download to focus on content pages.

Quality Assurance

The two-stage LLM analysis (Sonnet for classification + Sonnet QC for quality) ensures only high-quality, resolved cases with proper anonymization are indexed. Unresolved or low-confidence cases are discarded.

Incremental Ingestion

Gmail ingestion supports incremental loading with --offset:

# First batch (newest 300 threads)
uv run ingest-curated --max-threads 300

# Next batch (skip first 300, get next 300)
uv run ingest-curated --offset 300 --max-threads 300

# Continue incrementally
uv run ingest-curated --offset 600 --max-threads 300

Legacy Docs Ingestion

A separate ingestion pipeline for the legacy documentation site (support.cyanview.com), completely independent from the main CLORAG search engine.

Local Markdown Ingestion

Command: uv run ingest-legacy-docs [path] --fresh

Reads .md files directly from Docusaurus source directories, reconstructs URLs from file paths, and stores in the docusaurus_docs_legacy collection. Preserves original text — no RIO terminology transforms are applied.

Live Site Ingestion

Available via the Legacy Manage admin page:

  • Scan Sitemap — Compares live sitemap against indexed URLs, shows new pages for batch ingestion
  • Ingest Page — Ingest a single page by URL using Jina Reader + keyword extraction
  • Full Re-ingest — Re-crawl entire site using the standard Docusaurus pipeline
No Text Transforms

Legacy ingestion preserves original terminology (e.g., "RIO-Live" stays as-is). This is intentional — the legacy docs are for reviewing and updating the existing production documentation, not the new site.
