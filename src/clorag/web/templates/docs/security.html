Session-Based Auth

Admin routes are protected by session-based authentication:

  • Login via /admin/login page
  • Signed session cookies using itsdangerous
  • Cookie settings: HttpOnly, Secure, SameSite=Strict
  • Session expiry: 24 hours
Legacy Support

API calls can also authenticate via X-Admin-Password header for backward compatibility with scripts.

Brute Force Protection

Login attempts are protected with automatic lockout:

  • Threshold: 5 failed attempts per IP
  • Lockout Duration: 5 minutes
  • Rate Limit: 10 requests/minute to login endpoint
  • Timing-Safe Compare: Prevents timing attacks on password verification
Lockout Recovery

If locked out, wait 5 minutes. The lockout is per-IP and tracked server-side. Restarting the server will clear the lockout tracker.

XSS Protection

User-generated content is sanitized before rendering:

  • DOMPurify: All markdown→HTML conversions are sanitized with DOMPurify
  • escapeHtml(): Raw text is HTML-escaped before DOM insertion
  • SVG Allowlist: Mermaid diagrams use whitelisted SVG elements
  • No inline scripts: Content Security Policy considerations

Data Anonymization

Customer data in Gmail threads is anonymized before LLM processing:

Data TypeReplacementExample
Serial Numbers [SERIAL:XXX-N] SN12345 → [SERIAL:XXX-1]
Email Addresses [EMAIL-N] john@example.com → [EMAIL-1]
Phone Numbers [PHONE-N] +1-555-1234 → [PHONE-1]
IP Addresses [IP-N] 192.168.1.100 → [IP-1]

OAuth Token Security

Gmail OAuth tokens are encrypted at rest:

  • Encryption: Fernet symmetric encryption (AES-128-CBC)
  • Key Derivation: PBKDF2-HMAC-SHA256 with 480,000 iterations
  • Salt: Random 16-byte salt stored in data/.token_salt
  • Key Source: Derived from ADMIN_PASSWORD environment variable
  • File Permissions: Token files are chmod 600 (owner read/write only)
Backward Compatibility

The system automatically migrates unencrypted tokens. On first load after upgrade, plain JSON tokens are read and re-saved with encryption.

SQL Injection Prevention

  • Parameterized Queries: All SQL uses ? placeholders with separate values
  • Column Whitelist: Dynamic UPDATE queries validate columns against an allowlist
  • No String Concatenation: User input never concatenated into SQL strings

API Key Security

API keys are handled securely:

  • SecretStr Type: Pydantic's SecretStr prevents accidental logging
  • Environment Variables: Keys loaded from env vars, not config files
  • .env in .gitignore: Local env files excluded from version control

Security Configuration

# .env file settings
ADMIN_PASSWORD=strong_random_password_here    # Required, 16+ chars recommended
SECURE_COOKIES=true                           # Set false for local HTTP dev
QDRANT_API_KEY=your_qdrant_key               # Secures vector database

Security Best Practices

1. Rotate API keys periodically. 2. Use strong admin password (16+ chars, mixed case, numbers, symbols). 3. Monitor access logs for suspicious activity. 4. Keep dependencies updated. 5. Back up token salt file with encrypted tokens.
