Draft Auto-Reply System

Overview

The draft system automatically generates email replies for unanswered support threads using RAG search and Claude AI synthesis.

Pipeline Flow

┌─────────────────────────────────────────────────────────────────────┐
│                    DRAFT CREATION PIPELINE                          │
└─────────────────────────────────────────────────────────────────────┘

┌───────────────┐     ┌───────────────┐     ┌───────────────┐
│    GMAIL      │     │   ANALYZE     │     │    FILTER     │
│    FETCH      │────▶│   THREADS     │────▶│   PENDING     │
│  via API      │     │ ThreadAnalyzer│     │ no cyanview   │
└───────────────┘     └───────────────┘     └───────────────┘
                                                    │
        ┌───────────────────────────────────────────┘
        ▼
┌───────────────┐     ┌───────────────┐     ┌───────────────┐
│ CHECK DRAFTS  │     │  RAG SEARCH   │     │    CLAUDE     │
│  in Gmail     │────▶│ docs + cases  │────▶│   SYNTHESIS   │
│ skip existing │     │ hybrid query  │     │ draft content │
└───────────────┘     └───────────────┘     └───────────────┘
                                                    │
        ┌───────────────────────────────────────────┘
        ▼
┌───────────────┐
│ CREATE DRAFT  │
│  in Gmail     │
│ In-Reply-To   │
└───────────────┘

Admin Interface

Access the draft management UI at /admin/drafts:

  • Pending Threads - List of unanswered threads with expandable message history
  • Thread Timeline - Visual timeline showing all messages with Cyanview badges
  • Gmail Links - Direct links to open threads in Gmail
  • Preview Draft - View AI-generated response before creating
  • Create Draft - Create the draft reply in Gmail
  • Run Pipeline - Batch process all pending threads

CLI Command

# Process all unanswered threads (max 10)
uv run draft-support

# Preview drafts without creating
uv run draft-support --preview

# Process specific thread
uv run draft-support --thread THREAD_ID

# Increase limit
uv run draft-support --max 20

Google Groups Handling

The system correctly handles messages arriving via Google Groups:

  • Parses X-Original-Sender header for actual customer email
  • Falls back to Reply-To header when available
  • Strips "via Group" suffix from display names
  • Correctly identifies Cyanview vs customer messages
OAuth Scope

Draft creation requires gmail.compose scope. If using existing OAuth token, delete token_draft.json to re-authenticate with new scopes.

Performance Features

The camera database includes several performance optimizations:

  • FTS5 Full-Text Search - SQLite FTS5 virtual table with BM25 ranking and Porter stemming for fast search
  • TTL Cache - Thread-safe LRU cache (100 entries, 5-minute TTL) for query results
  • Connection Pool - 5-connection pool with WAL mode and 64MB SQLite cache
  • Covering Index - Optimized index for common list query patterns

Data Sources

Camera data is extracted from multiple sources:

  • Documentation - Automatic extraction during docs ingestion
  • Support Cases - Extraction from resolved support threads
  • Manual Entry - Admin interface for manual additions

Camera Browser

Related Cameras

The related cameras feature finds similar cameras based on:

  • Same device type (+3 similarity score)

  • Shared protocols (+1 per matching protocol)

Camera Merge (Deduplication)

Overview

The camera database accumulates duplicate entries from different sources where the same product has name variations: "C300 Mark II" vs "C300 MKII", "ILCE-FX6" vs "FX6", etc. The merge feature helps administrators find and consolidate these duplicates.

Finding Duplicates

Click "Find Duplicates" on /admin/cameras. The algorithm:

  • Name normalization — Lowercase, strip manufacturer prefix, replace synonyms (Mark II/MK2/MKII → mk2, Mark III/MK3 → mk3), strip " Head" suffix, remove spaces/hyphens/dots
  • Code model cross-reference — If camera A's code_model matches camera B's name (e.g., ILCE-FX6 ↔ FX6), they're grouped together
  • Union-find grouping — Overlapping groups are merged so transitive matches are caught

Groups are highlighted with alternating background colors and all cameras are auto-selected.

Merging Cameras

Select 2+ cameras via checkboxes (or use "Find Duplicates"), then click "Merge Selected":

  1. Choose the primary camera to keep (radio buttons)
  2. Optionally enter a custom name
  3. Review the merged preview (combined ports and protocols)
  4. Click "Merge" to execute

Merge Logic

Field TypeStrategy
Array fields (ports, protocols, controls, notes)Set union — all unique values combined
Scalar fields (manufacturer, code_model, device_type, URLs)Keep primary's value, fall back to first non-null from merge targets
ConfidenceMaximum across all cameras
NameCustom name if provided, otherwise primary's name

API Endpoints

MethodEndpointDescription
GET/api/admin/cameras/duplicatesFind groups of suspected duplicates
POST/api/admin/cameras/mergeMerge cameras (rate limited: 3/min)

Merge Request Body

    "primary_id": 42,
    "merge_ids": [87, 93],
    "custom_name": "Canon C300 Mark II"  // optional
}

FTS5 Full-Text Search

The database includes an FTS5 virtual table for fast full-text search across:

  • Subject line
  • Problem summary
  • Solution summary
  • Full document content
  • Keywords

Thread Cleaning

Raw threads are cleaned before storage to remove redundant content:

  • Quoted text - Lines starting with >
  • Reply headers - "On [date], [person] wrote:" patterns
  • Forwarded headers - "---------- Original Message ----------"
  • Email signatures - "Best regards", "Sent from my iPhone", etc.

Multi-language support includes English, French, and German reply header patterns.

Admin Interface

Access the support cases browser at /admin/support-cases:

  • Statistics - Total cases, breakdown by category/product/quality
  • Search - Full-text search with BM25 ranking
  • Filters - Filter by category or product
  • Detail View - Modal with tabs for Summary, Full Document, Raw Thread
  • Quality Stars - Visual display of resolution quality scores

Analytics System

Data Collected

Each search logs the following data:

  • Raw query text and source filter
  • Normalized query — after deterministic RIO terminology transforms (stored only when it differs from raw)
  • Rewritten query — LLM standalone rewrite for chat-mode follow-ups (stored only when a rewrite occurred)
  • Pipeline tag — web, chat, cli_agent, or mcp
  • Tool calls — per-invocation list for agent pipelines (tool name, query, result count)
  • Response time (milliseconds)
  • Number of results
  • Full LLM response text
  • Retrieved chunks (for replay without re-query)
  • Result scores (reranker scores when reranking enabled)
  • Source types per result (documentation, gmail_case, custom_docs)
  • Session ID (for conversation grouping)
  • Timestamp

Statistics Available

  • Total searches (with time range)
  • Average response time
  • Average results per query
  • Rerank coverage % — share of searches that went through the cross-encoder
  • Daily search volume chart
  • Source Insights — per-source win rate (% of searches ranked #1), avg top-5 reranker score, position mix at ranks 1–5
  • Recent conversations (grouped by session) with italic answer preview under each query
  • Recent searches (clickable to view details)
  • Search quality review — low-scoring queries via /api/admin/search-quality

Admin Interface

Access the analytics dashboard at /admin/analytics:

  • Time-filtered statistics (7d, 30d, all time)
  • Source Insights panel for retrieval quality by source
  • Click any search in Recent Searches or a conversation query to open the detail modal — shows raw / rewritten / normalized query variants, agent tool calls, full response, and retrieved chunks with scores

Performance Monitoring

Overview

The system includes comprehensive performance monitoring for the RAG pipeline with real-time metrics collection, automatic slow operation detection, and detailed statistics.

Metrics Collected

MetricTargetWarningDescription
embedding_generation<200ms>500msDense + sparse embedding generation (parallel)
vector_search<100ms>300msQdrant hybrid search with RRF fusion
total_search<500ms>1000msComplete search pipeline (embeddings + search + filtering)
llm_synthesis<2000ms>5000msTime to first token from Claude

Performance Optimizations

  • Parallel Embedding Generation - Dense (Voyage AI) and sparse (BM25) embeddings generated concurrently using asyncio.gather()
  • Query Embedding Cache - LRU cache (200 entries) for both dense and sparse query embeddings
  • Sparse Model Preloading - BM25 model loaded at server startup to eliminate cold start latency (~2-3s)
  • Async Voyage Client - Non-blocking API calls using voyageai.AsyncClient
  • Dynamic Prefetch Scaling - Prefetch limit scales with requested results (3x, max 50) for better RRF fusion
  • Connection Pooling - SQLite connection pools (5 connections) for camera and support case databases

Dark Mode

Overview

All admin pages support a dark theme. Click the moon/sun icon in the navbar to toggle. The preference is saved in localStorage and persists across sessions.

Behavior

  • First visit — Respects the OS-level prefers-color-scheme media query
  • Manual toggle — Overrides OS preference, saved in localStorage as clorag-theme
  • No flash — Theme is applied before DOMContentLoaded to prevent a flash of the wrong theme
  • Smooth transition — 350ms ease on background, color, border, and box-shadow properties

Technical Details

The dark mode is implemented via CSS custom properties on [data-theme="dark"]. All surface backgrounds, text colors, borders, shadows, tags, and alerts use semantic variables defined in admin.css. The ThemeToggle module in admin.js handles state management and injects the toggle button into the navbar dynamically (no template changes needed).

Animations

The admin UI includes subtle animations for a polished feel:

  • Page entrance — Fade-in with upward slide
  • Card stagger — Dashboard cards appear with staggered delays
  • Stat counters — Numbers count up from zero on page load
  • Modals — Scale-in with spring curve, animated close
  • Alerts — Slide down on appear, slide up on dismiss
  • Running badges — Pulse animation for active ingestion jobs

All animations are disabled when the user has prefers-reduced-motion: reduce set in their OS accessibility settings.

Answer Export

Users can export AI responses in multiple formats directly from the search results:

  • Markdown — Original formatted response with source links
  • Plain Text — Stripped formatting for pasting into emails or tickets
  • HTML — Rendered HTML for sharing or embedding
  • PDF — Downloadable document for offline use or printing

User Feedback

Thumbs up/down buttons appear on every AI answer after streaming completes:

  • Thumbs up — One-click positive feedback
  • Thumbs down — Opens optional text field for "What could be improved?"
  • Upsert logic — One vote per search (linked via search_id), can change your vote
  • Rate limited — 10 submissions per minute

Admin dashboard at /admin/analytics shows satisfaction rate, total feedback count, and recent feedback with comments.

Conversation Grounding

Multi-turn conversations use a strict grounding protocol to prevent answer contamination:

  • Intent resolution — History helps resolve follow-up intent (e.g., "and the FX6?" → "connect the FX6")
  • Fact isolation — Each answer is grounded exclusively in the current retrieved context — facts from previous answers are never mixed in
  • Enforcement — Message-level separator between history and new question, plus <conversation_grounding> rule in the synthesis prompt

Admin Messages

Admin-managed announcements displayed on the public index page in a collapsible "Latest Updates" panel:

  • 4 types — Info (blue), Warning (orange), Feature (green), Fix (red)
  • Expiration — Auto-hides expired messages, panel hidden when 0 active
  • Admin CRUD — Manage at /admin/messages

Legacy Docs System

A completely independent RAG system for the legacy documentation site at support.cyanview.com:

  • Separate collection — docusaurus_docs_legacy has no interaction with the main 3 collections
  • Public search — /legacy with streaming Claude synthesis, source links to .com (no URL rewriting)
  • Admin management — /legacy/manage (password-protected) for sitemap scanning, single-page ingest, full re-crawl
  • Help page — /legacy/help with usage documentation
  • Local markdown ingestion — uv run ingest-legacy-docs reads Docusaurus .md sources directly
  • No text transforms — Legacy content preserves original terminology (e.g., "RIO-Live" stays as-is)
  • Brown theme — Visual distinction from the main blue CLORAG UI
