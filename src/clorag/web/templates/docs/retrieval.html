Dense vs Sparse Embeddings

The system uses two complementary embedding types for optimal retrieval:

Aspect Dense (Voyage AI) Sparse (BM25)
Vector Type 1024 floats (continuous values) Millions of dimensions, mostly zeros
What It Captures Semantic meaning, concepts Exact keyword matches
Strength "RIO configuration" finds "RCP IP setup" "RIO" finds exactly "RIO"
Weakness May miss exact technical terms Misses synonyms/paraphrases
Best For Conceptual questions, how-to queries Product codes, model numbers, exact terms

Why BM25 Instead of Voyage for Sparse?

Voyage AI doesn't offer sparse embeddings. Their models (voyage-3, voyage-context-3) only produce dense vectors. BM25 via FastEmbed is the industry-standard choice for sparse/keyword matching:

  • Free - No API costs, runs locally
  • Fast - Local computation, no network latency
  • Proven - Classic TF-IDF based algorithm, decades of research
  • Complementary - Pairs perfectly with neural dense embeddings

Dynamic Score Thresholds

Search results are filtered using adaptive thresholds after reranking, using calibrated reranker scores (0-1). This is critical because RRF scores from Qdrant are uncalibrated and can exceed 1.0, making pre-rerank filtering unreliable.

Query LengthBase ThresholdRationale
≤2 words0.15Short queries need permissive matching
3-5 words0.20Medium specificity
>5 words0.25Specific queries can be strict

Technical terms (30+ terms including rio, rcp, visca, sdi, hdmi, ndi, srt, ptz, etc.) add +0.05 to the threshold. The system always returns at least 3 results regardless of threshold. A single shared implementation in core/retriever.py is used by both the CLI and web pipelines.

Source Diversity

When searching across all collections (hybrid mode), the top results can be dominated by a single collection (typically documentation). Source diversity enforcement ensures representation from each collection:

  • After merging and sorting by RRF score, the system checks if each collection is represented in the top results
  • If a collection is missing, its best result is inserted (replacing the lowest-scoring result) provided the score is at least 50% of the top result's score
  • This ensures support cases and custom docs appear alongside documentation when they have relevant content

Cross-Encoder Reranking

After hybrid search returns candidates, a cross-encoder reranker refines the relevance ranking for significantly improved results:

How It Works

Unlike bi-encoders (used for initial retrieval) that encode query and documents separately, cross-encoders process query+document pairs jointly. This enables much more accurate relevance prediction at the cost of speed—which is why we use it only on the top candidates.

┌──────────────────────────────────────────────────────────────────────┐
│                         TWO-STAGE RETRIEVAL                          │
│                                                                      │
│   Stage 1: Hybrid Search (fast, approximate)                         │
│   ┌─────────────────────────────────────────────────────────────────┐│
│   │  Query → Embed → Search Qdrant → RRF Fusion                     ││
│   │  Returns ~15 candidates (over-fetch 3x limit)                   ││
│   └─────────────────────────────────────────────────────────────────┘│
│                              │                                       │
│                              ▼                                       │
│   Stage 2: Reranking (slower, precise)                               │
│   ┌─────────────────────────────────────────────────────────────────┐│
│   │  For each candidate:                                            ││
│   │    relevance = rerank(query, document)                          ││
│   │  Sort by relevance score (0-1)                                  ││
│   │  Return top-K (default: 5)                                      ││
│   └─────────────────────────────────────────────────────────────────┘│
└──────────────────────────────────────────────────────────────────────┘

Configuration

# Environment variables
RERANK_ENABLED=true          # Enable/disable globally (default: true)
VOYAGE_RERANK_MODEL=rerank-2.5  # Model choice (default: rerank-2.5)
RERANK_TOP_K=5               # Results after reranking (default: 5)

Per-Query Control

Reranking can be disabled for individual queries where speed is more important than precision:

# In code
result = await retriever.retrieve(query, use_reranking=False)

# The agent tools always use reranking for best quality
Best Practice

Keep reranking enabled (default) for user-facing queries. The latency increase (~300ms) is negligible compared to LLM synthesis time, but the relevance improvement is significant.

GraphRAG Knowledge Graph

Overview

GraphRAG extends the traditional RAG system with a Neo4j knowledge graph that captures relationships between entities. This provides richer context for Claude synthesis by understanding how cameras, products, protocols, and issues are interconnected.

Graph Population

The knowledge graph is populated from Qdrant chunks using Claude Sonnet for entity extraction:

# Populate graph from all collections
uv run populate-graph

# Populate from specific collections only
uv run populate-graph --collections docusaurus_docs gmail_cases

# Limit chunks for testing
uv run populate-graph --max-chunks 100
Entity Extraction

Each chunk is analyzed by Claude Sonnet to extract entities and relationships. The process is idempotent - running it multiple times will merge new entities with existing ones.

Configuration

# Neo4j Configuration
NEO4J_URI=bolt://localhost:7687
NEO4J_USER=neo4j
NEO4J_PASSWORD=your_secure_password
NEO4J_DATABASE=neo4j  # Optional, defaults to "neo4j"

Local Development

The production Neo4j instance is bound to localhost only. For local development, use SSH tunneling:

# Create SSH tunnel to production Neo4j
ssh -L 7687:localhost:7687 root@cyanview.cloud -N -f

# Then set in local .env
NEO4J_URI=bolt://localhost:7687

Graceful Degradation

If Neo4j is not configured or unavailable:

  • The system falls back to standard RAG (vector search only)
  • No errors are thrown - graph enrichment is simply skipped
  • Check NEO4J_PASSWORD env var to enable GraphRAG
Graph Statistics

View current graph stats at /api/admin/graph/stats (requires authentication). Current production graph: ~5,500 nodes and ~3,600 relationships.

Performance Optimizations

Query Embedding Cache

Both dense (Voyage AI) and sparse (BM25) query embeddings are cached using a thread-safe LRU cache:

  • Cache size: 200 unique queries per embedding type
  • Benefits: Reduces Voyage AI API calls for repeated queries
  • Thread-safe: Safe for concurrent requests
  • Statistics: Hit rate tracking for monitoring

Paginated Document Listing

Custom knowledge documents use cursor-based pagination:

  • Incremental scrolling: Batches of 100 chunks instead of loading all
  • Early exit: Stops when enough documents collected
  • API response: Includes total count for pagination UI

Diagram Generation (Excalidraw Style)

Claude autonomously generates Mermaid diagrams when explaining integration scenarios, which are rendered in Excalidraw's hand-drawn style:

  • Camera connections - How to connect cameras to RIO/RCP/CI0/VP4
  • Network topology - IP setup and network architecture
  • Signal flow - Video, control, and tally routing
  • Multi-device setups - Complex integration scenarios

Diagrams use @excalidraw/mermaid-to-excalidraw to convert Mermaid syntax into Excalidraw's distinctive hand-drawn aesthetic, making technical diagrams more approachable and visually appealing.

Contextualized Embeddings

voyage-context-3 uses the /v1/contextualizedembeddings API (not /v1/embeddings) to encode each chunk together with its full document context. This produces richer semantic representations that understand each chunk's role within its parent document.

Document-Context Re-embedding

When chunk content is edited post-ingestion (e.g., terminology fixes), ALL sibling chunks from the same document are re-embedded together to maintain contextual consistency. Siblings are grouped by url (docs), thread_id (cases), or parent_doc_id (custom docs).

Conversation Grounding

Multi-turn conversations follow strict grounding rules to prevent answer contamination across turns:

  • Intent resolution — Conversation history helps resolve ambiguous follow-ups (e.g., "and the FX6?" → "how to connect the FX6")
  • Fact isolation — Each answer is grounded exclusively in the currently retrieved context. Facts from previous answers are never mixed into the current response.
  • Enforcement — A message-level separator is injected between history and the new question in synthesis.py. A <conversation_grounding> rule in the synthesis.web_layer prompt reinforces this boundary.

User Feedback

Every AI answer includes thumbs up/down buttons (enabled after streaming completes):

  • Thumbs up — One-click positive signal
  • Thumbs down — Opens an optional text field for improvement suggestions
  • Storage — search_feedback table with upsert per search_id (one vote per search)
  • Admin visibility — Satisfaction rate, total counts, and recent feedback with comments at /admin/analytics
  • Rate limit — 10 submissions per minute via POST /api/feedback
