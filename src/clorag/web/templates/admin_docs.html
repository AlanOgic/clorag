Cyanview AI Search - Technical Documentation

This document provides comprehensive technical documentation for administrators and developers working with the Cyanview AI Search platform.

System Architecture

Overview

Cyanview AI Search is a Multi-RAG (Retrieval-Augmented Generation) system that combines:

  • Documentation - Official Docusaurus support documentation
  • Support Cases - Curated and anonymized Gmail support threads
  • Camera Database - Structured camera compatibility information

Data Flow

User Query
    ↓
Embeddings Generation (Voyage AI voyage-context-3)
    ↓
Hybrid Search (Dense + Sparse vectors)
    ↓
RRF Fusion (Reciprocal Rank Fusion)
    ↓
Claude Haiku 4.5 Synthesis
    ↓
Streaming Response

Search System

Hybrid Search

The search system uses a hybrid approach combining semantic and keyword matching:

  • Dense vectors (1024-dim) - Voyage AI contextualized embeddings for semantic similarity
  • Sparse vectors - BM25 for exact keyword matching (important for model numbers)
  • RRF Fusion - Reciprocal Rank Fusion combines both results for optimal ranking

Collections

Two Qdrant collections store the indexed data:

CollectionContentVectors
docusaurus_docsDocumentation pagesdense + sparse
gmail_casesAnonymized support casesdense + sparse

Retrieval Settings

  • Retrieved chunks: 10 per collection
  • Used for synthesis: 8 top chunks
  • Max chunk size: 2000 characters

Conversation Sessions

The search system supports follow-up conversations with context retention:

  • Session Storage - Server-side LRU cache with max 1000 concurrent sessions
  • History Depth - Last 3 Q&A exchanges are kept for context
  • Session Timeout - 30 minutes of inactivity
  • Context Passing - Previous exchanges are included in Claude messages

Users can ask follow-up questions without repeating context. A "New" button resets the conversation.

Data Sources

Camera data is extracted from multiple sources:

  • Documentation - Automatic extraction during docs ingestion
  • Support Cases - Extraction from resolved support threads
  • Manual Entry - Admin interface for manual additions

Authentication

Session-Based Auth

Admin routes are protected by session-based authentication:

  • Login via /admin/login page
  • Signed session cookies using itsdangerous
  • Cookie settings: HttpOnly, Secure, SameSite=Strict
  • Session expiry: 24 hours
Legacy Support

API calls can also authenticate via X-Admin-Password header for backward compatibility with scripts.

Rate Limiting

  • Login attempts: 5/minute per IP
  • Camera create/update/delete: 5/minute per IP

API Reference

Search Endpoints

POST /api/search

Perform RAG search with Claude-powered synthesis.

{
  "query": "How to configure RIO IP address?",
  "source": "both",  // "docs", "gmail", or "both"
  "limit": 10,
  "session_id": null  // Optional: UUID from previous response for follow-ups
}

Response includes session_id to use in subsequent requests for follow-up conversations.

POST /api/search/stream

Streaming version with Server-Sent Events (SSE). First event sends {"type": "session", "session_id": "..."}.

Camera Endpoints (Public)

GET /api/cameras

List all cameras. Supports filters: manufacturer, device_type, port, protocol.

GET /api/cameras/search?q={query}

Search cameras by name or manufacturer.

GET /api/cameras/{id}

Get a single camera by ID.

Camera Endpoints (Admin)

POST /api/admin/cameras

Create a new camera entry. Requires authentication.

PUT /api/admin/cameras/{id}

Update an existing camera. Requires authentication.

DELETE /api/admin/cameras/{id}

Delete a camera entry. Requires authentication.

Analytics Endpoints

GET /api/admin/search-stats?days=30

Get search statistics. Requires authentication.

GET /api/admin/search/{id}

Get stored search details including response and chunks. Requires authentication.

Auth Endpoints

POST /api/admin/login

Authenticate and receive session cookie.

{ "password": "your_admin_password" }
POST /api/admin/logout

Clear session cookie.

GET /api/admin/session

Check if current session is valid.

Analytics System

Data Collected

Each search logs the following data:

  • Query text and source filter
  • Response time (milliseconds)
  • Number of results
  • Full LLM response text
  • Retrieved chunks (for replay without re-query)
  • Timestamp

Statistics Available

  • Total searches (with time range)
  • Average response time
  • Searches by source (docs/gmail/both)
  • Daily search volume chart
  • Popular queries (ranked)
  • Recent searches (clickable to view details)

Deployment

Docker Deployment

# Build and deploy
rsync -avz --exclude '.venv' --exclude 'data' . root@server:/opt/clorag/
ssh root@server "cd /opt/clorag && docker compose build && docker compose up -d"

Configuration

Ingestion Commands

# Ingest documentation
uv run ingest-docs

# Ingest Gmail support cases (curated)
uv run ingest-curated --max-threads 300

# Incremental ingestion
uv run ingest-curated --offset 300 --max-threads 300

Database Locations

  • data/clorag.db - Camera compatibility database
  • data/analytics.db - Search analytics database
Backup Recommendation

Regularly backup the SQLite databases in the data/ directory. These contain camera entries and search analytics that cannot be recovered from vector stores alone.
