Cyanview AI Search - Technical Documentation

This document provides comprehensive technical documentation for administrators and developers working with the Cyanview AI Search platform.

System Architecture

Overview

Cyanview AI Search is a Multi-RAG (Retrieval-Augmented Generation) system that combines:

  • Documentation - Official Docusaurus support documentation
  • Support Cases - Curated and anonymized Gmail support threads
  • Custom Knowledge - Admin-managed documents with full metadata
  • Camera Database - Structured camera compatibility information

Data Flow

User Query
    ↓
Embeddings Generation (Voyage AI voyage-context-3)
    ↓
Hybrid Search (Dense + Sparse vectors across 3 collections)
    ↓
RRF Fusion (Reciprocal Rank Fusion)
    ↓
Claude Sonnet 4.5 Synthesis + Mermaid Diagrams
    ↓
Streaming Response

Technology Stack

ComponentTechnology
Vector DatabaseQdrant
Graph DatabaseNeo4j (GraphRAG)
Dense EmbeddingsVoyage AI (voyage-context-3)
Sparse EmbeddingsFastEmbed BM25
LLM SynthesisClaude Sonnet 4.5

Diagram RenderingMermaid.js v11 (ESM)

DatabaseSQLite (cameras + analytics)
Session Managementitsdangerous (signed cookies)

Search System

Hybrid Search

The search system uses a hybrid approach combining semantic and keyword matching:

  • Dense vectors (1024-dim) - Voyage AI contextualized embeddings for semantic similarity
  • Sparse vectors - BM25 for exact keyword matching (important for model numbers)
  • RRF Fusion - Reciprocal Rank Fusion combines both results for optimal ranking

Collections

Three Qdrant collections store the indexed data:

CollectionContentVectors
docusaurus_docsDocumentation pagesdense + sparse
gmail_casesAnonymized support casesdense + sparse
custom_docsAdmin-managed knowledgedense + sparse

Retrieval Settings

  • Retrieved chunks: 10 per collection
  • Used for synthesis: 8 top chunks
  • Max chunk size: 2000 characters

Dense vs Sparse Embeddings

The system uses two complementary embedding types for optimal retrieval:

Aspect Dense (Voyage AI) Sparse (BM25)
Vector Type 1024 floats (continuous values) Millions of dimensions, mostly zeros
What It Captures Semantic meaning, concepts Exact keyword matches
Strength "RIO configuration" finds "RCP IP setup" "RIO" finds exactly "RIO"
Weakness May miss exact technical terms Misses synonyms/paraphrases
Best For Conceptual questions, how-to queries Product codes, model numbers, exact terms

Why BM25 Instead of Voyage for Sparse?

Voyage AI doesn't offer sparse embeddings. Their models (voyage-3, voyage-context-3) only produce dense vectors. BM25 via FastEmbed is the industry-standard choice for sparse/keyword matching:

  • Free - No API costs, runs locally
  • Fast - Local computation, no network latency
  • Proven - Classic TF-IDF based algorithm, decades of research
  • Complementary - Pairs perfectly with neural dense embeddings
Key Code Paths

app.py:stream_search() - orchestrates the flow
vectorstore.py:hybrid_search() - RRF fusion
embeddings.py:embed_query() - dense vectors
sparse_embeddings.py:embed_query() - BM25 vectors
graph/enrichment.py - Neo4j context
app.py:_synthesize_response() - Claude streaming

Mermaid Diagram Generation

Claude autonomously generates Mermaid.js diagrams when explaining integration scenarios:

  • Camera connections - How to connect cameras to RIO/RCP/CI0/VP4
  • Network topology - IP setup and network architecture
  • Signal flow - Video, control, and tally routing
  • Multi-device setups - Complex integration scenarios

Diagrams are rendered client-side using Mermaid.js v11 ESM modules with Cyanview color theming (navy blue nodes, cyan connections).

Graph Population

The knowledge graph is populated from Qdrant chunks using Claude Haiku for entity extraction:

# Populate graph from all collections
uv run populate-graph

# Populate from specific collections only
uv run populate-graph --collections docusaurus_docs gmail_cases

# Limit chunks for testing
uv run populate-graph --max-chunks 100
Entity Extraction

Each chunk is analyzed by Claude Haiku to extract entities and relationships. The process is idempotent - running it multiple times will merge new entities with existing ones.

Configuration

GraphRAG requires Neo4j to be configured in the environment:

# Neo4j Configuration
NEO4J_URI=bolt://localhost:7687
NEO4J_USER=neo4j
NEO4J_PASSWORD=your_secure_password
NEO4J_DATABASE=neo4j  # Optional, defaults to "neo4j"

Local Development

The production Neo4j instance is bound to localhost only. For local development, use SSH tunneling:

# Create SSH tunnel to production Neo4j
ssh -L 7687:localhost:7687 root@cyanview.cloud -N -f

# Then set in local .env
NEO4J_URI=bolt://localhost:7687

Graceful Degradation

If Neo4j is not configured or unavailable:

  • The system falls back to standard RAG (vector search only)
  • No errors are thrown - graph enrichment is simply skipped
  • Check NEO4J_PASSWORD env var to enable GraphRAG

Knowledge Graph Explorer

The admin UI at /admin/graph provides a visual interface for exploring the knowledge graph:

  • Overview: Total node and relationship counts by type
  • Entity Browser: Browse entities by type with pagination and search
  • Relationship Browser: View and filter relationships between entities
  • Relationship Management: Edit (change type) or delete relationships directly from the UI
Graph Statistics

View current graph stats at /api/admin/graph/stats (requires authentication). Current production graph: ~5,500 nodes and ~3,600 relationships.

Performance Optimizations

The RAG system includes several optimizations for better performance and cost efficiency:

Query Embedding Cache

Both dense (Voyage AI) and sparse (BM25) query embeddings are cached using a thread-safe LRU cache:

  • Cache size: 200 unique queries per embedding type
  • Benefits: Reduces Voyage AI API calls for repeated queries
  • Thread-safe: Safe for concurrent requests
  • Statistics: Hit rate tracking for monitoring

Dynamic Score Thresholds

Search results are filtered using adaptive thresholds based on query characteristics:

Query LengthBase ThresholdRationale
≤2 words0.15Short queries need permissive matching
3-5 words0.20Medium specificity
>5 words0.25Specific queries can be strict

Technical terms (rio, rcp, firmware, etc.) add +0.05 to the threshold. The system always returns at least 3 results regardless of threshold.

Paginated Document Listing

Custom knowledge documents use cursor-based pagination:

  • Incremental scrolling: Batches of 100 chunks instead of loading all
  • Early exit: Stops when enough documents collected
  • API response: Includes total count for pagination UI

Quality Assurance

The two-stage LLM analysis (Haiku for speed + Sonnet for quality) ensures only high-quality, resolved cases with proper anonymization are indexed. Unresolved or low-confidence cases are discarded.

Incremental Ingestion

Gmail ingestion supports incremental loading with --offset:

# First batch (newest 300 threads)
uv run ingest-curated --max-threads 300

# Next batch (skip first 300, get next 300)
uv run ingest-curated --offset 300 --max-threads 300

# Continue incrementally
uv run ingest-curated --offset 600 --max-threads 300

Custom Knowledge Base

Overview

The Knowledge Base allows administrators to add custom documents that are included in RAG search alongside documentation and support cases. This is useful for:

  • Internal technical notes not published to documentation
  • Firmware release notes and known issues
  • Troubleshooting guides from field experience
  • Temporary information with expiration dates
  • FAQ entries and best practices

Document Metadata

  • Title - Display name for the document
  • Content - Full text content (auto-chunked for embedding)
  • Category - One of the 9 categories above
  • Tags - Comma-separated keywords for filtering
  • URL Reference - Optional link to source or related page
  • Expiration Date - Optional date when document should be excluded from search
  • Notes - Internal admin notes (not included in search)

Admin Interface

Access the Knowledge Base manager at /admin/knowledge:

  • Add Document - Two input modes:
    • Paste Text - Enter content directly in a textarea
    • Upload File - Drag-and-drop or browse for .txt, .md, or .pdf files
  • Browse Documents - List all documents with filtering by category and expired status
  • Edit Document - Update content and metadata (triggers re-embedding)
  • Delete Document - Remove document and all its chunks from the vector store

File Upload

The file upload feature supports:

FormatProcessing
.txtPlain text, UTF-8 with latin-1 fallback
.mdMarkdown text, UTF-8 with latin-1 fallback
.pdfText extraction using pypdf library
  • Maximum file size is determined by server configuration
  • Minimum content length: 10 characters after extraction
  • Title auto-populated from filename if not specified
  • All other metadata (category, tags, etc.) can be set during upload
Chunking and Embedding

Documents are automatically split into 1000-character chunks with 100-character overlap, then embedded using voyage-context-3 (dense) and BM25 (sparse). All chunks from a document share the same metadata and parent_doc_id for grouping.

Bulk Import CLI

For importing multiple documents at once, use the import-docs CLI command:

# Preview what would be imported (dry run)
uv run import-docs ./your-folder --category pre_sales --dry-run

# Import all documents with category and tags
uv run import-docs ./your-folder --category pre_sales --tags "presales,integration"

# Import with default category (other)
uv run import-docs ./your-folder

Supported file types: .txt, .md, .pdf

  • Recursively scans the folder for supported files
  • Auto-generates titles from filenames (underscores/hyphens become spaces)
  • Applies the same category and tags to all imported documents
  • Use --dry-run to preview without importing

Chunk Editor

Overview

The Chunk Editor provides direct access to browse, search, and edit individual chunks in the vector database. This is useful for:

  • Inspecting what content is actually indexed
  • Correcting inaccuracies in ingested content
  • Fine-tuning chunk metadata (tags, titles)
  • Deleting outdated or incorrect chunks
  • Debugging search relevance issues

Admin Interface

Access the Chunk Editor at /admin/chunks:

  • Collection Selector - Switch between docusaurus_docs, gmail_cases, and custom_docs
  • Search - Full-text search within the selected collection
  • Browse - Paginated list of all chunks with metadata preview
  • View/Edit - Expand a chunk to see full content and edit in-place
  • Delete - Remove individual chunks from the vector store

Human Modified Flag

When you edit a chunk, the human_modified flag is automatically set to true. This protects your edits from being overwritten by future ingestion runs. Re-ingesting the same source will not update human-modified chunks.

Re-Embedding on Edit

When chunk text is edited:

  1. New dense (Voyage AI) and sparse (BM25) embeddings are generated
  2. The chunk is updated in Qdrant with new vectors
  3. The human_modified flag is set to prevent overwrite
  4. Changes are immediately reflected in search results

API Endpoints

Chunk operations are available via REST API:

  • GET /api/admin/chunks - List chunks with pagination and search
  • GET /api/admin/chunks/{id} - Get single chunk details
  • PUT /api/admin/chunks/{id} - Update chunk content/metadata
  • DELETE /api/admin/chunks/{id} - Delete a chunk
Deleting vs Editing

Prefer editing over deleting when correcting information. Deleted chunks leave gaps in document coverage. If a chunk contains completely wrong info, edit it to correct the information rather than removing it.

Conversation Sessions

The search system supports follow-up conversations with context retention:

  • Session Storage - Server-side LRU cache with max 1000 concurrent sessions
  • History Depth - Last 3 Q&A exchanges are kept for context
  • Session Timeout - 30 minutes of inactivity
  • Context Passing - Previous exchanges are included in Claude messages

User Interface

  • Chat-style layout - User questions appear as bubbles, AI answers in cards
  • Inline follow-up input - After each answer, an input field appears below for the next question
  • Natural flow - Conversation stacks vertically, scrolling to show new exchanges
  • New conversation - Floating button resets session and clears history

Google Groups Handling

The system correctly handles messages arriving via Google Groups:

  • Parses X-Original-Sender header for actual customer email
  • Falls back to Reply-To header when available
  • Strips "via Group" suffix from display names
  • Correctly identifies Cyanview vs customer messages
OAuth Scope

Draft creation requires gmail.compose scope. If using existing OAuth token, delete token_draft.json to re-authenticate with new scopes.

API Connection Guide

Quick Start

Connect to the CLORAG API using any HTTP client. All responses are JSON.

Base URL

Production: https://cyanview.cloud
Local Dev:  http://localhost:8080

Authentication Methods

1. Session Cookie (Recommended for Browser)

Login via the web form or API to receive a signed session cookie:

# Login
curl -X POST https://cyanview.cloud/api/admin/login \
  -H "Content-Type: application/json" \
  -d '{"password": "your_admin_password"}' \
  -c cookies.txt

# Use cookie for subsequent requests
curl https://cyanview.cloud/api/admin/search-stats \
  -b cookies.txt

2. X-Admin-Password Header (For Scripts)

Include the admin password in the header for programmatic access:

curl -X POST https://cyanview.cloud/api/admin/cameras \
  -H "Content-Type: application/json" \
  -H "X-Admin-Password: your_admin_password" \
  -d '{"name": "Canon C70", "manufacturer": "Canon"}'

Session Security

FeatureImplementationPurpose
Signed Cookies itsdangerous.URLSafeTimedSerializer Prevent tampering and forgery
HttpOnly Flag httponly=True Prevent JavaScript access (XSS protection)
Secure Flag secure=True (production) HTTPS-only cookie transmission
SameSite Strict samesite="strict" Prevent CSRF attacks
Session Expiry 24 hours Limit exposure window

Brute Force Protection

Login attempts are protected with automatic lockout:

  • Threshold: 5 failed attempts per IP
  • Lockout Duration: 5 minutes
  • Rate Limit: 10 requests/minute to login endpoint
  • Timing-Safe Compare: Prevents timing attacks on password verification
Lockout Recovery

If locked out, wait 5 minutes. The lockout is per-IP and tracked server-side. Restarting the server will clear the lockout tracker.

XSS Protection

User-generated content is sanitized before rendering:

  • DOMPurify: All markdown→HTML conversions are sanitized with DOMPurify
  • escapeHtml(): Raw text is HTML-escaped before DOM insertion
  • SVG Allowlist: Mermaid diagrams use whitelisted SVG elements
  • No inline scripts: Content Security Policy considerations

Data Anonymization

Customer data in Gmail threads is anonymized before LLM processing:

Data TypeReplacementExample
Serial Numbers [SERIAL:XXX-N] SN12345 → [SERIAL:XXX-1]
Email Addresses [EMAIL-N] john@example.com → [EMAIL-1]
Phone Numbers [PHONE-N] +1-555-1234 → [PHONE-1]
IP Addresses [IP-N] 192.168.1.100 → [IP-1]

OAuth Token Security

Gmail OAuth tokens are encrypted at rest:

  • Encryption: Fernet symmetric encryption (AES-128-CBC)
  • Key Derivation: PBKDF2-HMAC-SHA256 with 480,000 iterations
  • Salt: Random 16-byte salt stored in data/.token_salt
  • Key Source: Derived from ADMIN_PASSWORD environment variable
  • File Permissions: Token files are chmod 600 (owner read/write only)
Backward Compatibility

The system automatically migrates unencrypted tokens. On first load after upgrade, plain JSON tokens are read and re-saved with encryption.

SQL Injection Prevention

  • Parameterized Queries: All SQL uses ? placeholders with separate values
  • Column Whitelist: Dynamic UPDATE queries validate columns against an allowlist
  • No String Concatenation: User input never concatenated into SQL strings

API Key Security

API keys are handled securely:

  • SecretStr Type: Pydantic's SecretStr prevents accidental logging
  • Environment Variables: Keys loaded from env vars, not config files
  • .env in .gitignore: Local env files excluded from version control

Security Configuration

# .env file settings
ADMIN_PASSWORD=strong_random_password_here    # Required, 16+ chars recommended

Security Best Practices

1. Rotate API keys periodically. 2. Use strong admin password (16+ chars, mixed case, numbers, symbols). 3. Monitor access logs for suspicious activity. 4. Keep dependencies updated. 5. Back up token salt file with encrypted tokens.

Authentication

Session-Based Auth

Admin routes are protected by session-based authentication:

  • Login via /admin/login page
  • Signed session cookies using itsdangerous
  • Cookie settings: HttpOnly, Secure, SameSite=Strict
  • Session expiry: 24 hours
Legacy Support

API calls can also authenticate via X-Admin-Password header for backward compatibility with scripts.

Rate Limiting

  • Login attempts: 5/minute per IP
  • Camera create/update/delete: 5/minute per IP

API Reference

Search Endpoints

POST /api/search

Perform RAG search with Claude-powered synthesis.

{
  "query": "How to configure RIO IP address?",
  "source": "both",  // "docs", "gmail", or "both"
  "limit": 10,
  "session_id": null  // Optional: UUID from previous response for follow-ups
}

Response includes session_id to use in subsequent requests for follow-up conversations.

POST /api/search/stream

Streaming version with Server-Sent Events (SSE). First event sends {"type": "session", "session_id": "..."}.

Get a single camera by ID.
